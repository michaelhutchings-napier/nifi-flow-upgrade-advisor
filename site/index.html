Migrate Deliberately

Upgrade Apache NiFi flows without guessing what will break.

nifi-flow-upgrade-advisor is a customer-facing CLI for NiFi flow migration. It inspects source artifacts, explains version-specific caveats, applies only deterministic safe rewrites, and validates target readiness before you import.

What customers actually need

Teams do not need a mystery upgrader. They need a tool that can tell them whether a selected source flow can move to a selected target version, highlight deprecated or removed components, preserve the configuration that can be preserved, and stop before doing anything ambiguous.

Confidence-first migration

  • Version-pair rule packs grounded in official Apache NiFi migration guidance
  • Deterministic rewrites only for mechanically safe changes
  • Live target validation against NiFi version, extension inventory, and process-group readiness
  • Readable Markdown and machine-friendly JSON reports for humans and CI
  • Support for legacy `flow.xml.gz` analysis as well as modern JSON artifacts
  • Explicit publish targets for filesystems, Git-backed registries, and NiFi Registry
  • Secrets-safe by design: references stay, missing sensitive values stay external

How it works

Analyze the source

Load `flow.json.gz`, versioned snapshots, NiFi Registry exports, Git-backed registry directories, or legacy `flow.xml.gz` and compare them against the selected target version.

Rewrite only what is safe

Rename properties, replace component types, and apply other deterministic transforms only when the rule pack makes the mapping explicit and mechanically safe.

Validate the destination

Check the target NiFi version, runtime extension inventory, and the readiness of the destination process group before you attempt an import or update in production.

Publish deliberately

Push the reviewed result into a filesystem handoff, a Git-backed registry tree, or an external NiFi Registry version history without turning publish into hidden runtime behavior.

Why this is safer than “just export and import”

Standard export and import workflows do not tell you which processors are deprecated, which properties changed, or whether the target process group is even a valid update candidate. This tool makes that impact visible before you touch production.

It also keeps the right boundary around secrets: NiFi intentionally does not export sensitive values in versioned flow definitions, so the tool preserves references and reports what needs rehydration instead of pretending it can recover hidden values. 

nifi-flow-upgrade analyze \
  --source ./flow.json.gz \
  --source-version 1.27.0 \
  --target-version 2.0.0 \
  --rule-pack ./examples/rulepacks/nifi-1.27-to-2.0.official.yaml \
  --output-dir ./out

nifi-flow-upgrade rewrite \
  --plan ./out/migration-report.json \
  --output-dir ./out

nifi-flow-upgrade validate \
  --input ./out/rewritten-flow.json.gz \
  --target-version 2.0.0 \
  --target-api-url https://nifi.example.com \
  --target-process-group-id 1234-abcd \
  --output-dir ./out

nifi-flow-upgrade publish \
  --input ./out/rewritten-snapshot.json \
  --input-format versioned-flow-snapshot \
  --publisher nifi-registry \
  --registry-url https://registry.example.com \
  --registry-bucket-name customer-a \
  --registry-flow-name orders \
  --output-dir ./out

Built for real delivery teams

Customer review first

Every migration produces reviewable reports, so platform teams and flow owners can agree on the change before publish.

CI ready

JSON reports make it easy to gate promotion on blocked or manual-change findings without parsing terminal output.

External by design

The tool stays outside runtime platforms like NiFi-Fabric so migration logic does not turn into hidden controller behavior.
