Custom Components

Teach the advisor about your own processors and services.

Private NiFi components are supported through private rule packs. The safest pattern is to start with review-focused rules, then promote them to assisted or automatic rewrites only when your team owns the migration contract.

Back to site Open source doc

Recommended tiers

  • auto-fix for mechanically safe internal renames
  • assisted-rewrite when the tool can scaffold the target shape but a person still needs to review it
  • manual-change for migrations that depend on libraries, credentials, or architecture

Good first uses

  • package and type renames
  • property renames and copied values
  • bundle-coordinate updates
  • internal service replacements where your team controls both sides
class: assisted-rewrite
actions:
  - type: replace-component-type
    from: com.acme.nifi.CustomerApiFetcher
    to: com.acme.nifi.HttpCustomerFetcher
  - type: copy-property
    from: Legacy URL
    to: Remote URL
  - type: set-property-if-absent
    property: HTTP Method
    value: GET

Why assisted rewrite exists

Some internal migrations are too specific for blind auto-fix but too repetitive to leave completely manual. Assisted rewrite gives you a reviewable middle tier: the tool scaffolds the target shape in a separate output artifact and keeps the final decision visible.
