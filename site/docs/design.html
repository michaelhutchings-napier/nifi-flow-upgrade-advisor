Design

External-first, confidence-first NiFi migration tooling.

The advisor stays outside runtime platforms on purpose. Migration analysis and rewrite logic should be reviewable, explicit, and safe to run offline.

Back to site Open source doc

Design choices

  • desktop-first user experience with a CLI engine underneath
  • explicit rule-pack input instead of scraping docs at runtime
  • safe deterministic rewrites only
  • read-only validation against target NiFi and extension manifests
  • publish as a deliberate handoff, not hidden runtime mutation

Why separate from platform repos

Keeping the engine in its own repo prevents migration logic from turning into hidden controller behavior. Platform repos can link to it, pin versions, and integrate it in CI without owning the migration engine directly.

Secret handling

NiFi does not export sensitive values in a way this tool can recover. The advisor preserves references and flow structure, but secret rehydration belongs to Parameter Contexts, Parameter Providers, Vault, Kubernetes Secrets, or similar environment-local systems.

Supported targets

The tool can validate and publish to filesystems, Git-backed registry directories, and NiFi Registry. Live target validation checks version, runtime extensions, and process-group readiness.
